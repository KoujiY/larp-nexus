# 玩家端設計規範

**版本**：1.0
**建立日期**：2026-03-24
**設計參考**：`.impeccable.md`（品牌個性、色彩系統、技術約束）
**對應計畫**：`PHASE_D_PLAN.md` D-4 玩家側頁面設計規範

---

## 總體設計方向

玩家端的核心情感目標：**「這是我的角色」**。

從掃描 QR Code 的那一刻開始，每個畫面都應強化代入感——不是在操作一個遊戲 app，而是在翻開一本屬於自己角色的書。

### 佈局原則

- **手機直向優先**（375px 基準），桌面為次要
- 所有主要操作按鈕（使用道具、使用技能、回應對抗）必須落在**拇指自然觸碰區**（畫面下半 50%）
- 每個畫面的視覺重量應從上（裝飾/識別）往下（操作）遞增
- 最大內容寬度：`max-w-2xl`（672px），保持手機閱讀的舒適感

### 色彩使用規範

| 情境 | Token | 用途 |
|------|-------|------|
| 角色主色 | `text-primary` / `bg-primary` | 標題高亮、主要 CTA 按鈕 |
| 成功/完成 | `text-success` / `bg-success` | 任務完成、道具使用成功 |
| 警告/冷卻 | `text-warning` / `bg-warning` | 冷卻中、道具耗盡警告 |
| 危險/錯誤 | `text-destructive` | 對抗失敗、數值過低（≤25%） |
| 資訊提示 | `text-info` / `bg-info` | 狀態說明、被揭露的隱藏資訊 |
| 環境（Runtime）| `bg-env-runtime` | 遊戲進行中模式 banner |
| 環境（預覽）| `bg-muted` | 預覽模式 banner |

## 畫面規範

### 1. 公開頁面（`/g/[gameId]`）

**情感目標**：玩家的第一印象——在踏入遊戲世界之前的入口。應帶有儀式感，讓人感受到「即將進入某個不平凡的故事」。

**畫面狀態**：
- `loading`：頁面資料讀取中
- `active`：遊戲進行中（可進入角色卡）
- `inactive`：遊戲尚未開始或已結束

#### 佈局結構（由上至下）

```
┌─────────────────────────────┐
│  [劇本封面圖 / 裝飾性橫幅]     │  ← 全寬，aspect-ratio: 16/9，bg-primary/10 作底
│  若無圖，以奇幻紋理 CSS 代替    │
├─────────────────────────────┤
│  劇本名稱（text-4xl bold）    │  ← 居中，留白充裕
│  副標（若有：公開描述節錄）     │  ← text-muted-foreground
├─────────────────────────────┤
│  ▸ 世界觀 / 前導故事（主要內容）│  ← prose 排版，支援 whitespace-pre-wrap
├─────────────────────────────┤
│  [狀態區塊]                  │
│  ○ 進行中：顯示「進入遊戲」CTA  │  ← bg-primary text-primary-foreground，佔滿寬度
│  ○ 未開始：說明文字（等待開始） │
│  ○ 已結束：說明文字（感謝參與） │
└─────────────────────────────┘
```

#### 設計細節

- **無劇本封面圖時**：用 CSS 製作奇幻氛圍底色（`bg-primary/5` + 細微紋理 or 漸層），而非留白的空 div
- **「進入遊戲」按鈕**：大尺寸（`h-14`），寬度填滿（`w-full`），帶有輕微進入動畫（Framer Motion fade-up）
- **遊戲未開始時**：用一個精緻的「計時沙漏」或「蠟燭」圖示（Lucide `Hourglass`），加說明文字，不顯示任何 CTA

#### 已知問題（M-01）
目前視覺設計簡陋，需在 D-5b 實作時完整設計此畫面。

### 2. 角色卡解鎖畫面（PIN Unlock）

**情感目標**：像是拿起一個信封、拆開封蠟——帶有儀式感和期待感，不是在「輸入密碼」。

**畫面狀態**：
- `idle`：等待輸入
- `verifying`：驗證中（loading）
- `error`：PIN 錯誤
- `preview_unlocked`：僅 PIN 解鎖，進入預設模式
- `full_unlocked`：PIN + 遊戲代碼解鎖，進入完整模式

#### 佈局結構

```
┌─────────────────────────────┐
│  [角色名稱]（text-2xl）       │  ← 居中，帶淡入動畫
│  角色初始字（若有圖則不顯示）  │
├─────────────────────────────┤
│  [解鎖說明]                  │  ← 簡短、帶儀式感的文字
│  "輸入你的角色 PIN 以進入"    │
├─────────────────────────────┤
│  ┌──────────────────────┐   │
│  │  PIN 輸入欄           │   │  ← 大字體、居中，OTP 風格輸入
│  └──────────────────────┘   │
│                              │
│  [進階選項，摺疊]             │
│  ▸ 輸入遊戲代碼（完整互動模式）│
├─────────────────────────────┤
│  [解鎖按鈕]                  │  ← 全寬，h-14，在鍵盤出現後仍可見
└─────────────────────────────┘
```

#### 設計細節

- **PIN 輸入**：建議使用大號數字鍵盤樣式（`inputmode="numeric"`），`type="password"` 隱藏輸入
- **遊戲代碼欄**：預設摺疊，避免新手困惑。展開時帶 Framer Motion slide-down
- **錯誤狀態**：輸入框 shake 動畫（Framer Motion `keyframes`），邊框變 `border-destructive`，不使用 alert
- **新手引導**（M-02 要求）：首次進入時，在 PIN 欄下方顯示一行提示：「PIN 由 GM 提供，如不清楚請詢問主持人」

#### 互動流程

```
輸入 PIN → 點解鎖
  ├─ 只輸入 PIN → 進入預設模式（唯讀）
  └─ 輸入 PIN + 遊戲代碼 + 遊戲進行中 → 進入完整模式（互動）
```

### 3. 角色卡（主畫面）

這是玩家使用頻率最高的畫面，分三個子狀態共享同一 URL。

#### 3a. 角色識別區（所有模式共用）

位於畫面最上方，在 Tab 切換過程中始終可見（固定高度，不隨 Tab 內容滾動）。

```
┌─────────────────────────────────────────┐
│  [角色圖片 / 初始字佔位]                  │  ← aspect-16/9，全寬（或正方形 + 圓角）
├─────────────────────────────────────────┤
│  [模式 Banner]                           │  ← CharacterModeBanner（預覽 / 遊戲進行中）
├─────────────────────────────────────────┤
│  角色名稱（text-3xl bold）  [通知鈴鐺]   │  ← 名稱佔左，通知佔右
│  個性描述（text-muted，1行）             │
└─────────────────────────────────────────┘
```

**角色圖片無圖時的佔位設計**：
- 背景：`bg-primary/10`，帶微弱圖樣（純 CSS，不依賴圖片）
- 中央顯示：角色名稱第一個字，`text-6xl font-bold text-primary/40`
- 不使用通用 User icon（太泛用，沒有身份感）

#### 3b. Tab 導航列

```
┌──────┬──────┬──────┬──────┬──────┐
│ 道具 │ 技能 │ 資訊 │ 數值 │ 任務 │
└──────┴──────┴──────┴──────┴──────┘
```

- **固定於角色識別區下方**（`sticky top-0`），滾動時不消失
- Tab 高度：`h-12`（達 WCAG 44px），圖示 + 文字標籤（手機上文字隱藏）
- 啟用中的 Tab：底部 `border-b-2 border-primary`，文字 `text-primary`
- Tab 切換：Framer Motion `AnimatePresence` fade 轉場

#### 3c. 預設模式（唯讀）

Banner 顯示 `CharacterModeBanner`（灰色背景，「預覽模式」字樣）。

所有「使用」按鈕改為 ghost 樣式並顯示鎖頭圖示，hover 時 tooltip 說明「需以完整模式進入才可使用」。

不顯示對抗流程、道具轉移等進階操作入口。

#### 3d. 完整模式（可互動）

Banner 顯示 `CharacterModeBanner`（`bg-env-runtime` 琥珀金底，「遊戲進行中」字樣）。

所有操作按鈕啟用。

### 4. 道具頁（Tab: items）

**情感目標**：玩家的工具箱，要感覺像翻開背包，而非查看庫存清單。

#### 佈局結構

```
┌─────────────────────────────────────────┐
│  道具數量提示（「共 N 件道具」）           │  ← text-sm text-muted-foreground
├─────────────────────────────────────────┤
│  ┌────────────────────────────────────┐ │
│  │ [道具卡片]                          │ │  ← 全寬 list，垂直堆疊
│  │  左：圖示（圖片 or Package icon）   │ │
│  │  右：名稱 + 描述（1行截斷）         │ │
│  │       使用限制 badge + 效果 badge   │ │
│  │  右下：[使用] 按鈕                  │ │  ← 固定在卡片右下角，拇指可達
│  └────────────────────────────────────┘ │
└─────────────────────────────────────────┘
```

#### 道具卡片狀態

| 狀態 | 視覺表現 |
|------|---------|
| 可使用 | 正常，`[使用]` 按鈕 `variant="default"` |
| 冷卻中 | `opacity-70`，按鈕顯示倒計時（格式化為 `Xh Xm Xs`） |
| 已耗盡 | `opacity-40`，按鈕 `disabled`，顯示「已耗盡」 |
| 預覽模式 | 按鈕 `ghost` + 鎖頭圖示 |

#### 空狀態

```
┌─────────────────────────────────────────┐
│         [Package icon, h-12, opacity-30]│
│         目前沒有道具                     │
└─────────────────────────────────────────┘
```
`bg-muted/20 rounded-xl py-16`，無邊框，無虛線。

### 5. 技能頁（Tab: skills）

與道具頁結構相似，以下列出差異：

- **技能卡片**：以 `Zap` icon 為主圖示，無消耗品概念
- **冷卻顯示**：與道具相同的格式化倒計時
- **檢定類型 badge**：「對抗檢定」/ 「隨機對抗檢定」/ 「隨機檢定」，badge `variant="outline"`
- **空狀態**：使用 `Zap` icon

### 6. 資訊頁（Tab: info）

包含：公開資訊（所有人可見）+ 隱藏資訊（遊戲中逐步揭露）。

#### 隱藏資訊被揭露時的視覺強調

新揭露的隱藏資訊應有視覺區隔：
- 邊框：`border-info/30`
- 背景：`bg-info/5`
- 左側裝飾條：`border-l-2 border-info`
- 右上角標籤：「已揭露」badge（`variant="outline"` + `text-info`）

這個設計讓玩家一眼看出哪些是新獲得的資訊。

### 7. 數值頁（Tab: stats）

**StatsDisplay 進度條設計**（已實作，確認規範）：

| 百分比 | 進度條色彩 | 語義 |
|--------|-----------|------|
| > 75% | `bg-success` | 健康 |
| 26–75% | `bg-info` | 正常 |
| ≤ 25% | `bg-destructive` | 危急 |

**活躍效果（ActiveEffectsPanel）**：
- 每個效果顯示為小卡片，帶倒計時進度條
- 倒計時接近結束時（< 30s）進度條顏色轉為 `bg-warning`，帶輕微 pulse 動畫

### 8. 任務頁（Tab: tasks）

**任務狀態視覺**：

| 狀態 | 樣式 |
|------|------|
| 進行中 | 正常顯示 |

| 隱藏任務（已揭露）| 同隱藏資訊，`border-info/30 bg-info/5` |

### 9. 對抗流程 Dialog（C-02 修復規範）

這是 C-02 的設計解法，**不在 D-5b 之前實作**，但此規範是實作依據。

#### 攻擊方等待 Dialog

```
┌─────────────────────────────┐
│  等待防守方回應...           │
│  [角色名稱]                  │
│  [進度條或旋轉動畫]          │
├─────────────────────────────┤
│  ⚠ 對抗進行中，請稍待        │
│  [取消並放棄本次行動]  ← ghost│  ← 提供退出路徑
└─────────────────────────────┘
```

「取消並放棄」應有二次確認（confirm dialog），防止誤觸。取消後，本次道具/技能使用視為未執行（server 端需對應 API）。

#### 防守方回應 Dialog

```
┌─────────────────────────────┐
│  ⚔ 你受到了攻擊              │
│  攻擊方：[名稱 or 隱匿]      │
│  使用：[道具/技能名稱]        │
├─────────────────────────────┤
│  選擇回應方式：              │
│  ○ 使用技能/道具回應         │  ← 預設選項
│  ○ 以基礎數值直接回應        │  ← 快捷選項，避免被困
├─────────────────────────────┤
│  [確認回應]  ← 全寬主按鈕    │
└─────────────────────────────┘
```

**關鍵設計決策**：
- 不允許完全關閉（遊戲機制需要），但提供「以基礎數值直接回應」的快捷路徑，讓玩家不需要翻找道具/技能也能脫困
- ESC 鍵不關閉，但 Dialog 頂部需顯示角色名稱，讓玩家確認自己的狀態

#### 遊戲結束 Dialog

- 顯示後 **5 秒允許關閉**（按鈕倒計時顯示），不強制鎖定
- 關閉後自動執行 relock 流程

## 動畫規範

| 情境 | 動畫 | 時長 |
|------|------|------|
| 畫面初始進場 | fade-in + slide-up (y: 8px → 0) | 250ms |
| Tab 切換 | fade (opacity 0→1) | 180ms |
| Dialog 開啟 | scale (0.96→1) + fade | 200ms |
| 錯誤狀態 | shake (x: ±4px × 3) | 300ms |
| 道具使用成功 | 短暫 scale-up (1→1.02→1) | 200ms |
| 進度條 | width transition | 500ms ease-out |

## 無障礙（a11y）基線

- 所有互動元件具備 `aria-label`（中文）
- 所有圖示 icon 配合周圍文字或 `aria-label`，不依賴 icon 單獨傳達語義
- 最小觸控目標：44×44px（Tab、按鈕、通知入口均已達標）
- 顏色對比度：文字 ≥ 4.5:1，大文字 ≥ 3:1（WCAG 2.1 AA）
- 鍵盤可導航：所有操作支援 Tab 鍵聚焦 + Enter/Space 觸發

## Stitch Prompt 要點

使用此規範生成 Stitch prompt 時，需包含：

1. **使用者情境**：「站立走動中、光線條件不穩定、單手操作手機」
2. **情感目標**：對應畫面的情感目標（如上方各畫面所述）
3. **色彩系統**：深午夜藍背景 + 琥珀金主色，Tailwind CSS 4 oklch token
4. **禁止項目**：無漸層紫色、無 Bootstrap 感、無通用 SaaS 設計語言
5. **重點畫面**：建議從「角色卡完整模式 - 道具頁」開始，這是使用頻率最高的畫面
